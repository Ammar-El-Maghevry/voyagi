# Database Conventions

## Naming and Types

- Database identifiers use `snake_case`; enum labels use the architecture's uppercase values.
- Internal relational identifiers use `bigint generated always as identity`. Externally exposed booking, payment, ticket, and commission identifiers use UUIDs.
- All timestamps use `timestamptz`. Monetary values use `numeric(12,2)` and three-letter uppercase currency codes.
- Fixed lifecycle states use PostgreSQL enums. JSONB is limited to layouts, policy/configuration objects, event metadata, and audit value snapshots.
- Phone/contact values use `varchar(21)` and a constrained international form: an optional `+` followed by 8 to 20 digits. The extra character preserves the `+` when all 20 digits are present.

## Integrity

- Foreign keys use `ON DELETE RESTRICT` for operational and financial history. Mutable records may use `SET NULL` for optional actors; immutable event, audit, and payment evidence retains actor references with `RESTRICT`.
- Composite foreign keys carry `company_id` through tenant-owned relationships, preventing cross-company buses, routes, branches, trips, bookings, maintenance records, and commissions.
- Active seats are unique by `(trip_id, seat_number)`. Released and cancelled rows remain as history and no longer block reuse.
- A seat layout is either a flat JSON array of canonical labels or an object with a `seat_numbers` array. Its labels must be distinct and match `total_seats`; reservations must reference one of those labels.
- Booking idempotency keys, payment internal/provider references, ticket identifiers, and agent-booking commissions are database-enforced unique values.
- Prices and cancellation policy are snapshots. Current route prices and company settings never rewrite existing trips or bookings.
- Booking references are generated by the backend in `VYG-YYYYMMDD-XXXXXX` form, for example `VYG-20260717-8F3D2A`, and remain database-unique.
- Tickets store only a one-way `qr_token_hash`. Raw QR bearer tokens must never be persisted or included in logs.
- Every table containing monetary amounts carries a three-letter currency code, defaulting to `MRU`.

## Security and Lifecycle

- RLS is enabled on every application table. Authenticated users receive read-only access filtered by company, branch, ownership, or manager scope; only profile name and phone are directly editable.
- Clients receive no direct insert/delete grants for sensitive tables. The backend uses its trusted server connection and owns all transactions.
- `trip_events` and `audit_logs` are append-only. Bookings and financial records reject physical deletion.
- `booking_events` is an append-only booking timeline; corrections are represented by later events, never by changing history.
- Reference data is retired using `is_active`, `deleted_at`, or `archived_at` rather than deletion.

## Migration Practice

- All schema changes are ordered migrations under `supabase/migrations` and must pass `supabase db reset` and `supabase db lint`.
- Production data is never placed in migrations. `supabase/seed.sql` contains reproducible local development data.
- A migration already applied to a shared environment is immutable; corrections are made with a new migration.
